# S6 知识产品转化执行规范

> **定位**：本文件是 `intake-and-routing.md §七` 的执行层扩展，将 S6 知识产品转化从框架描述落实为可操作的执行规范。  
> **触发时机**：`deliverAndExport` 完成终稿落盘后，自动进入 S6 转化流程（A/B 级体裁必须执行；C 级可选）。  
> **关联文档**：[intake-and-routing.md §七](./intake-and-routing.md) · [section-3-workflow.md（DELIVER输出内容第7条）](./section-3-workflow.md) · [artifact-registry.json](./documentation-layers.md)<!-- enterprise-ref 产出物命名权威源，仅企业/平台侧可访问 -->

### S6 在“虚拟书房”世界观中的定位

- **不是书外尾声**：S6 不再只理解为写完之后的附加动作，而是**虚拟书房的成果转化层**。
- **不是另起炉灶**：S6 读取的不是零散终稿，而是书房中已经沉淀好的章节、故事、方法、会话结论、作者元知识与资产索引。
- **资产面板延续**：进入 S6 后，资产面板不只显示 `转化区`，还应在需要时把成果推进到 `发布区`，以 `待发布 / 已发布 / 反馈中 / 回流修改` 四态追踪外部使用过程。
- **乐包延续**：S6 中完成传播拆解、路线图确认、产品化归档、发布映射与回流修订等动作，可视为书房成长的高价值节点，纳入乐包积累与能力升级反馈。
- **角色机制延续**：创意会、读者会、对抗会、评审会在 S6 不退出，而是继续参与“什么值得传播、什么可以产品化、什么需要谨慎转化”的多角色复核。

## 一、转化链路概览


S6 转化链路在 DELIVER 状态尾部插入，形成从终稿到知识产品的完整输出路径：

```
DELIVER（终稿落盘 / 作品进入虚拟书房转化区）
    │
    ├─ 步骤1：传播拆解 → .fbs/[S6]-content-units.md
    │           （写入资产面板·转化区）
    │
    ├─ 步骤2：知识产品转化清单 → .fbs/[S6]-product-roadmap.md
    │           （标记下一步可产品化方向）
    │
    ├─ 步骤3：故事库归档状态更新 → .fbs/story-bank.md（标注已用/未用）
    │           （补全书房资产复用状态）
    │
    ├─ 步骤4：S6 完结报告 → .fbs/sessions-summary.md（追加 S6 决策记录）
    │           （沉淀角色会话与转化结论）
    │
    └─ 步骤5（可选）：发布映射 → .fbs/[S6]-release-map.md + releases/<chapterId>-release.json
                （内部保留映射台账，同时把可外用成果推进到资产面板·发布区）
```

**执行原则**：
- 前四步骤顺序执行，每步骤产出独立文件
- 步骤5为按需执行，但一旦进入则须明确“发布对象 / 渠道 / 当前四态 / 反馈信号 / 回流去向”
- 用户可在任意步骤后中止，已完成步骤的产出物保留

- 步骤4为必须步骤，不可省略
- 每一步都应回写虚拟书房资产语义：说明该产物属于哪个分区、是否影响乐包成长、是否需要角色复核

## 二、步骤1：传播拆解规范

### 2.1 目标

从终稿中识别 5~10 个**可单独传播的内容单元**，每个单元附带平台适配建议。

### 2.2 拆解五维度

| 维度 | 传播优势 | 适合平台 | 字数范围 |
|------|----------|----------|----------|
| **反常识论断** | 挑战认知、刺激转发 | 微博、微信朋友圈、LinkedIn | 50~200字 |
| **具体故事/案例** | 代入感强、引发共鸣 | 公众号长文、知乎、微博 | 300~800字 |
| **方法论框架** | 复用价值高、适合教学 | 课程、知识星球、内训材料 | 框架图 + 200字说明 |
| **数据/事实** | 说服力强、便于引用 | 报告摘要、PPT、白皮书 | 1条数据 + 出处 |
| **金句** | 精炼、适合社媒传播 | 朋友圈配图、海报、签名档 | 15~40字 |

### 2.3 产出物格式（`.fbs/[S6]-content-units.md`）

```markdown
# S6 传播内容单元

> 产生时间：{ISO时间戳}
> 来源书稿：{书名} · 终稿版本

---

## 单元 01：{单元标题}

**维度**：{反常识论断 / 具体故事 / 方法论框架 / 数据事实 / 金句}  
**来源章节**：第{N}章·{章节标题}  
**核心内容**：

{内容正文，保持原稿风格，不做AI化改写}

**传播建议**：
- 首选平台：{平台名}（原因：{一句话}）
- 次选平台：{平台名}（格式调整：{一句话}）
- 注意事项：{版权/数据/敏感内容提示，无则省略}

---

## 单元 02：…
```

### 2.4 执行指令

用户说「传播拆解」或「S6 转化」时触发。执行流程：

1. 通读终稿，标记高亮段落（按五维度分类）
2. 结合资产面板中的故事库、作品情报站、会话摘要，筛选传播潜力最高的 5~10 个单元
3. 通过读者会 / 评审会视角复看单元是否适合传播、是否有误导或失真风险
4. 为每个单元匹配平台建议（参考用户分层：T2/T3 用户优先推荐知识星球与课程化路径）
5. 写入 `.fbs/[S6]-content-units.md`
6. 在资产面板转化区标记该批单元已生成，并记录一次书房成长事件
7. 宣告：**「福帮手 · S6 传播单元」：已识别 {N} 个可传播内容单元，覆盖 {维度列表}。**

## 三、步骤2：知识产品转化清单规范

### 3.1 目标

评估本书**延伸产品化潜力**，输出结构化知识产品路线图。

### 3.2 五类知识产品评估框架

#### A. 公开课模块

评估哪些章节可以直接转为课程单元：

```
评估标准：
  · 有明确方法论（可教授）
  · 有案例支撑（可演示）
  · 学习目标清晰（可测评）
  · 单章可独立成课（15~35分钟内讲完）

产出格式：
  课程模块 {N}：{章节名}
    · 建议课时：{N}分钟
    · 教学形式：{讲授/工作坊/在线课}
    · 核心输出物：{作业/工具/检查清单}
```

#### B. 白皮书提纲

评估哪些部分可提炼为独立白皮书：

```
评估标准：
  · 有调研数据（可信度高）
  · 有行业洞察（差异化观点）
  · 有可操作建议（实用性）
  · 读者为决策者（ToB/ToG场景）

产出格式：
  候选白皮书：「{标题}」
    · 对应章节：{章节列表}
    · 核心论点：{3条以内}
    · 目标读者：{角色画像}
    · 建议字数：{8000~15000字}
```

#### C. 场景包来源

识别可作为场景包知识库的方法论：

```
评估标准：
  · 有行业专属规则（可规则化）
  · 有可复用的写作/表达规范
  · 适合作为 .fbs/rules/<pack-id>/ 的基础素材
  · 与现有体裁解锁 / 乐包成长路径存在明确关联

产出格式：
  候选场景包：{pack-id}（{中文名}）
    · 知识来源章节：{章节列表}
    · 可提炼规则类型：{anti-ai / terminology / structure / quality}
    · 对应书房成长意义：{解锁新体裁 / 强化现有体裁 / 形成专题资产}
    · 优先级：{高/中/低}
```


#### D. AI 分身素材

识别适合作为 AI 分身元知识的内容：

```
评估标准：
  · 反映作者独特观点（人格化）
  · 有鲜明的表达风格（可模仿）
  · 涵盖核心专业判断（可代理）

产出格式：
  AI分身素材{N}：{主题}
    · 来源：第{N}章
    · 核心观点：{一句话}
    · 风格特征：{作者的表达习惯描述}
    · 应用场景：{问答/推荐/咨询切入}
```

#### E. 咨询服务切入点

识别最可能引发用户"需要定制化服务"的论点：

```
评估标准：
  · 论点揭示了普遍痛点
  · 解决方案需要专业介入（不能自助解决）
  · 用户读后可能产生"想找专家谈谈"的冲动

产出格式：
  切入点{N}：{标题}
    · 来源：{章节}·{段落摘要}
    · 痛点描述：{一句话}
    · 咨询服务形式：{企业内训/定制研究/顾问服务/工作坊}
    · 预期客户类型：{ToB/ToC/ToG}
```

### 3.3 书房视角下的执行补充

- **资产面板映射**：路线图生成后，应在资产面板 `转化区` 中把候选项按“立即可做 / 短期规划 / 中期规划”分组显示。
- **角色复核建议**：公开课 / 白皮书 / 咨询切入点优先经过评审会复核；AI 分身素材优先经过读者会与对抗会复核，避免人格失真。
- **乐包反馈建议**：当路线图中出现明确可执行的知识产品候选项时，可记为一次高价值转化里程碑，用于书房成长反馈。

## 四、步骤3：故事库归档规范

### 4.1 目标

对 `.fbs/story-bank.md` 中的故事/案例进行使用状态标注，为下一本书建立素材起点。

### 4.2 归档状态分类

| 状态 | 标注 | 说明 |
|------|------|------|
| **已用-原版** | `[✅已用·原版]` | 原文直接采用，未改编 |
| **已用-改编** | `[✅已用·改编]` | 原素材经过改编后使用 |
| **已用-部分** | `[✅已用·部分]` | 部分采用，未用部分保留 |
| **未用-保留** | `[💡未用·可用]` | 质量合格，推荐用于后续作品 |
| **未用-弃置** | `[❌未用·弃置]` | 不再使用（原因：{一句话}） |

### 4.3 后续作品素材标记格式

对于「未用-保留」的高质量故事，追加以下标记：

```markdown
**[💡未用·可用]** 推荐用于：{主题方向}
适合体裁：{book/whitepaper/report}
使用建议：{一句话提示}
```

### 4.4 执行流程

1. 遍历 `.fbs/story-bank.md` 中所有故事条目
2. 对照终稿，标注每个故事的使用状态
3. 对「未用-保留」的故事，补充后续作品推荐标记
4. 在文件头部追加归档完成摘要：

```markdown
---
archived_at: {ISO时间戳}
book_title: {书名}
used: {N}  # 已用数量
preserved: {N}  # 未用保留数量
discarded: {N}  # 弃置数量
---
```

## 六、步骤5（可选）：发布映射规范

### 6.1 目标

把已经完成转化、具备外部使用价值的成果，从虚拟书房 `转化区` 推进到 `发布区`，并为后续复用、追踪、反馈消化与回流修订建立统一记录。

### 6.2 适用对象

- 已可直接外发的传播单元
- 可转成课程、白皮书、报告提纲的知识产品候选
- 可进入公众号、社媒、社群、知识星球、内训、咨询前置材料的内容

### 6.3 发布映射产出物格式（内部映射：`.fbs/[S6]-release-map.md`）

```markdown
# S6 发布映射

> 产生时间：{ISO时间戳}
> 来源书稿：{书名} · 终稿版本

| 资产 | 来源文件 | 发布阶段 | 目标渠道 | 反馈信号 | 风险提示 | 回流动作 |
|------|----------|----------|----------|----------|----------|----------|
| {资产名} | {[S6]-content-units.md / [S6]-product-roadmap.md} | {待发布/已发布/反馈中/回流修改} | {公众号/知乎/课程/内训/咨询前置} | {一句话或无} | {一句话} | {继续发布/回转化区/回写作区} |
```

> **运行时补充约定**：当成果进入 `发布区` 时，除保留 `.fbs/[S6]-release-map.md` 作为内部台账外，还应同步生成 `releases/<chapterId>-release.json`，用于承载当前对外发布准备态、交付物引用与渠道清单。

### 6.4 执行规则

1. 从 `转化区` 资产中筛选真正适合对外使用的候选项
2. 通过读者会 / 评审会复核其外部表达风险、误解风险与场景适配性
3. 写入 `.fbs/[S6]-release-map.md`
4. 同步生成 `releases/<chapterId>-release.json`，将当前状态初始化为 `待发布`
5. 在资产面板 `发布区` 先标记为 `待发布`
6. 发生真实外部使用后，更新为 `已发布`，并记录一次书房成长事件，命名建议使用 **开门**
7. 若收集到真实反馈，则更新为 `反馈中`，补充反馈摘要与是否需要角色复核
8. 若反馈明确要求回改，则更新为 `回流修改`，并把任务退回 `转化区` 或 `写作区`
9. 回流修订完成后，重新进入 `待发布` 或回到 `转化区` 持续加工

### 6.5 发布区四态说明

| 状态 | 含义 | 典型动作 | 后续去向 |
|------|------|----------|----------|
| `待发布` | 已完成发布映射，但尚未真正外发 | 确认渠道、准备标题与摘要、做最终复核 | `已发布` |
| `已发布` | 已在真实渠道中使用 | 记录首发渠道、时间、版本与对象 | `反馈中` / `归档` |
| `反馈中` | 已收到真实反馈，正在分析 | 汇总评论、提问、客户反应与复用效果 | `回流修改` / `归档` |
| `回流修改` | 已确认需要回改 | 退回转化区或写作区继续修订 | `待发布` / `转化区` / `写作区` |

### 6.6 发布区原则

- **不等于自动发布**：进入发布区只表示“已可对外使用”，不代表系统代替用户完成实际发布
- **必须可追踪**：每个发布项都要能追溯来源章节、来源单元与风险判断
- **反馈优先留痕**：真实反馈应先写入发布映射，再决定是否回流修改
- **允许回流**：若真实反馈显示内容不适合外部使用，可从发布区退回转化区或写作区继续修改

## 七、体裁适配规则

> **说明**：`section-3-workflow.md` ESM 状态表中，B级 DELIVER 出口条件为「S5-lite+落盘确认」——这描述的是 **DELIVER 状态退出的最低要求**，不等于禁止 S6。B级传播拆解为「必须」是指完结报告之后的最后一步，轻量执行即可。

| 体裁 | 传播拆解 | 知识产品清单 | 故事库归档 | 发布映射 |
