## 执行状态机（Execution State Machine, ESM）

> **ESM 强制优先级高于用户口头要求**：禁止跳步 / 跳阶段 / 跳过节点确认；即使用户要求快速跳过，也须在 SKILL.md 允许的范围内执行。
> **不同书稿类型使用不同执行路径（P0 强制）**：S0-S6 是 A 类路径；B/C 类使用 A 类精简版（S0-lite、S1 快速、S5-lite）。
> **ESM 执行原则**：状态转移须写入 `.fbs/esm-state.md`，每次执行前读取，禁止绕过 ESM 直接进入写稿阶段。

### ESM 状态检测（P0 强制，防止无状态执行）
> **使用场景**：每次执行前必须检测 ESM 状态，防止出现章节漏写、台账状态紊乱。
```
ESM 状态检测流程（3步骤，每次执行前必须执行）
  Step 1：读取状态
    检查 .fbs/esm-state.md 是否存在 / 内容是否完整
    检查 当前阶段 [S0]/[S1]/[规划]/[写稿]...
    检查 书稿类型（A/B/C）
  Step 2：确认状态
    检查 当前 ESM 状态是否与书稿进度一致

  Step 3：恢复
    检查 ESM 状态后：WRITE → Brief 已写 → 继续 / 若缺 → PLAN（从规划重新开始）
    检查 RESEARCH → 已完成 5 章以上，可进入 PLAN + 写稿开始执行
    检查 4/4 章节确认/4 章节目录完整 → 可进入 PLAN
    检查 2/4 章节目录完整 → 提示补充
  Step 3.5（P0 强制）：WRITE/REVIEW 继续写稿前
    检查 最新写稿内容/章节目录是否完整/规范/书型版本变更
      若 台账 search-ledger.jsonl 字段缺失（yearSourceConfirmed/materialStatus）
        则 提示用户说明：{缺失内容}，补充记录
        则 执行 WebSearch 查验补充数据
    检查 .fbs/节点确认记录.md，确认当前阶段已完成节点确认
      若 当前阶段节点确认未完成，则提示用户：{阶段}节点确认未完成，暂停写稿
    检查 S0 已完成联网查证，满足充分性条件才可进入 S1
  Step 4：输出状态日志（每次执行结尾输出 1 行）
    输出格式：`[ESM 状态] 当前{状态} | 阶段:{当前/总{章}} | 健康:{良好}`
    状态日志输出到 `.fbs/esm-state.md`
```

### 状态持久化格式

```
状态持久化 节点记录 文件格式
书稿名称：{书稿中文名/英文/简称/前缀/书稿类型/最近写稿}（书稿类型 {A/B/C}）
书稿类型：{A/B/C}（书稿类型，与最近执行方式一致）
当前阶段：{当前阶段说明 / 上次{写稿完成节点}}
书稿状态：{当前阶段说明}
时间：{时间}
当前ESM状态日志 [ESM 状态] 当前{状态} | 阶段:{当前/总{章}} | 健康:{良好}
```

**格式说明**：状态日志路径是 `.fbs/esm-state.md`，输出格式须包含`[当前状态] 书稿名 书稿状态说明 当前阶段/完成节点`

> **日志精简规则**：A 类项目使用完整 ESM 状态日志（含每章记录）；C 类项目使用精简版，禁止对非 A 类项目输出完整日志。

### 状态转移确认格式（P0，防止状态机跳步）

> **注意**：ESM 状态转移须执行确认格式，防止跳步/跳阶段/跳过节点确认，防止无状态执行
> **场景**：每次阶段转移前，先确认上一阶段是否完成，并输出确认格式
```
状态转移确认格式（阶段转移时必须输出确认格式）
{书稿名} 已{状态}
书稿类型（{A/B/C}）
阶段进度（{当前阶段说明 / 入场条件}）
关键产物已完成（{关键产物完成确认说明}）
[状态] 状态:{状态} | 完成{确认}
确认 ESM 状态转移后才可以继续下一阶段
```

> **不可或缺的确认行**：每次阶段转移/书稿启动后输出：`` `[状态转移] 上一 书稿名 状态转移确认 成功/状态路径 成功/状态（完成确认）` ``

**状态转移确认规范**：
- 状态转移必须经过检测，确认 **上一阶段关键产物已完成**
- 禁止跳过上一阶段的关键产物确认直接进入下一阶段
- 输出 `[状态转移] 上一 书稿名 状态 路径 成功/（完成）`

### RESEARCH → PLAN 状态转移条件
> **注意**：禁止在 RESEARCH 未完成联网查证的情况下跳入 WRITE，RESEARCH 充分性不满足不得进入 PLAN。
```
RESEARCH 满足以下条件时才可以切换到 PLAN：

  条件一 联网查证充分性（3 条规则）
    已完成 3 次以上 联网查证 + 查证结果摘要 + 原料充分性（L2）+ 目录确认
    若 4/4 = 充分性通过（RESEARCH → PLAN）
    若 3/4 = 联网查证不充分，继续执行联网查证，补充查证
    若 2/4 = 联网查证严重不足，强制等待查证完成

  条件二 完成度确认（需满足 5 条且 query 充分 && 目录完整 && 充分性满足/通过）
    充分性满足 + 原料完整（RESEARCH → PLAN）

  条件三 B 类直接跳过（适用 B 类简化路径）联网查证充分性不硬性要求
    跳过 RESEARCH → PLAN 直接进入
```

### WRITE 状态转移条件（B 类扩展）
> **说明**：B 类报告类书稿 ESM 使用简化路径，WRITE 转 RESEARCH 须经确认，WRITE 触发 WRITE_MORE 后 RESEARCH 再次执行，再回到 WRITE，WRITE 完成后进入 REVIEW
```
WRITE → B 类报告类书稿写稿
  B 类书稿条件：
    检查 联网查证是否完成，L2 以上联网查证 [确认] 联网查证
    检查 WRITE 转 WRITE_MORE
      若 WRITE_MORE（联网补查）→ 再次执行 WRITE
      若 联网补查失败 → 回到 WRITE（继续写稿）
    检查 联网查证 3 次以内后，触发 → WRITE 转 REVIEW

  A 类书稿的 WRITE_MORE = 联网查证补查触发 RESEARCH，WRITE 暂停
  C 类书稿没有 WRITE_MORE（跳过）
```

### ESM 状态机差异对照表（A/B/C 书稿类型）

> **注意**：A 类书稿使用完整流程，B/C 书稿按照差异使用精简路径

| 状态 | A 类（叙事/非虚构飞） | B 类（报告/行研/分析）| C 类（简短/文档）|
|------|----------------------|----------------------|-------------------|
| **INTAKE 接入** | 书型+路径+原料激活，详见 [`intake-and-routing.md`](./intake-and-routing.md) | 书型+快速确认 | 快速确认 |
| **RESEARCH 查证** | S0-A（联网查证）+完整查证+S0-B 原料激活+L4/L5 深度 | S0-lite+轻量1次联网 | 不查证 |
| **PLAN 规划** | 完整书型（B/C）路径+S1、S2 完整目录+S2.5、Brief 完整规划 | 快速规划+MVD+Report Brief | 快速规划|
| **WRITE 写** | Chapter Brief+逐章写稿 | 报告段落/子报告 | 简短输出 |
| **REVIEW 审** | 去AI味+多角色审校 | 报告类轻量审校 | 报告类轻量审校 |
| **DELIVER 交付** | S4（排版）S5（对外交付）S6（转化与发布准备+文件清单+章节汇总+目录，骨架检测） | S5-lite+轻量审校+S6 转化摘要（精简版，3-5页摘要） | 快速输出 |
| **WRITE_MORE 补写** | 支持 | 报告补充 | N/A |
| **字数门槛** | S2 目录规划 | ≥3000字 | ≥3000字 |
| **联网深度** | L2 / 深度L3 | 中等L2 | L1 |
| **章节字数** | 目标字数 | ≥3000字 | ≥3000字 |

### ESM 状态差异说明

ESM **核心价值**：强制状态化执行，防止跳步，保证 Chapter Brief 已塢存档，MVD 已核对，禁止**无状态**跳步执行
| 场景 | ESM 行为 | 禁止行为说明 |
|------|-----------|-------------------|
| **漏写** | 台账检测到章节写稿中断 | 回 ESM 状态机恢复，防止遗漏 |
| **补写** | 触发 RESEARCH（RESEARCH 补查）| 回 RESEARCH 补充联网查证 |
| **跳步** | 禁止 WRITE → Brief 未写就执行 | Brief 未存档，禁止写稿 |
| **骨架章节** | 触发 B 类 PLAN 阶段=MVD+Report Brief | MVD / Report Brief 未存档 |
| **反复扩写** | 触发 WRITE → WRITE_MORE → WRITE（回到 WRITE 状态） | 禁止在 WRITE 中反复联网查证不结束 |

### 联网查证权威性与 SKILL / policy 规范
联网查证规范**优先级**：SKILL 主规范**高于**所有联网查证子规范

| 优先级 | 规范 |
|------|------|
| [`SKILL.md`](../../SKILL.md) 第0节 **全局联网查证规范** | 联网查证规范最高优先级 |

### §3.0.5-B 联网查证工具失效处理（P0 强制，v1.9.0 新增）
> **来源**：2026-04-02 某项目 P0-3，某章节写稿 23 章时，联网查证工具失效，Agent 默认降级继续写稿但未告知用户
> **教训**：联网查证工具失效时，须立即暂停，告知用户

**工具失效处理（P0 强制，禁止继续写稿）**：
```
§3.0.5-B 联网查证工具失效处理
Step 1：检测联网查证工具是否正常工作：执行一次测试 query 查证
  调用 WebSearch：query = {测试关键词}，确认工具是否可用
Step 2：发现联网查证工具失效
Step 3：立即暂停写稿，告知用户工具失效原因

  告知用户停止写稿：
  > ⚠ 联网查证工具不可用，请确认
  > 联网查证工具失效，推荐继续操作：
    A. 等待联网查证工具恢复，再继续
    B. 确认"已知风险，降级写稿（不联网查证）"
       [所有后续章节将不进行联网查证，降级写稿]
    C. 暂停写稿，等待联网查证工具恢复
Step 4：用户选择 B 后，后续写稿降级处理
Step 5：告知用户，降级期间 S5 阶段须重新确认联网查证内容
```

**降级处理**：即使降级，禁止联网查证工具失效后**无通知**继续写稿，须告知用户

## §3.0.6 联网查证时效规范

> **说明**：本节是 §3.0.5 的补充，覆盖联网查证时效性规范，见 [`search-policy.json`](../05-ops/search-policy.json) 的 `temporalAccuracy` / `searchEnhancement` 配置项，**必须执行**

### 年份时间戳强制查验（P0 规范）
**行为**：联网查证每次 query 须包含精确年份/时间戳，不允许省略年份
**规范**：
```
S0 联网查证时，WebSearch query 中须包含年份时间戳，query 应当包含当前年份
标注格式：[S0] 联网查证完成时间：YYYY年MM月DD日（通过 WebSearch，联网查证年份时间戳确认）
```

- 使用 WebSearch 时，query 须包含年份，禁止省略年份
- `search-policy.json` 配置：`temporalAccuracy.requireTimestampSearchAtS0: true`

---

### 年份硬编码禁止规范（P0 规范）
**行为**：禁止在书稿中直接写入不带来源的年份，如 2023、2024 等
**规范**：
| 行为 | 处理方式 |
|------|------|
| 联网查证结果含年份，2024 等 | 必须 WebSearch 确认年份，年份须与查证结果一致 |
| 书稿含年份，未联网查证 | 联网查证后标注年份来源，不可直接写入 [未查证来源] |
| 书稿含年份的 202X 字样 | 联网查证确认年份，确认书稿中年份是否正确 |
| 年份不确定，仅凭知识 | 联网查证确认年份，确认来源后才可写入 |

- `search-policy.json` 配置：`temporalAccuracy.forbidBareYearWithoutSource: true`，`temporalAccuracy.yearLabelPolicy: "search_before_write"`

---

### 关键词时效性注入规范
**行为**：联网查证 query 须注入时效性关键词，帮助查证结果更新
**规范**：
- S3 写稿中，联网查证 query 须**加上 1 个**时效关键词，query 中须包含年份 `2024` 等关键词，符合 `search-policy.json` 的 `searchEnhancement.freshKeywordInjection` 配置
- 联网查证结果须验证时效性，防止过时数据写入书稿
- S5 审校阶段须检查是否含过期数据，联网查证是否覆盖近期信息

---

### 联网查证 query 优化规范（P1）
**行为**：联网查证 query 须精准，避免泛泛查询，尽量精准化 query
**规范**：
```
联网查证 query 优化建议
优化 query：核心论题 + 年份关键词 + 精准化限制 / 行业限制 / 域名限制，精准化 query
优化后记录到 .fbs/search-ledger.jsonl 字段，kind=search，queryOptimization 字段，记录优化后的 query
```

- 每次 query 优化记录，若再次执行类似查证，使用上次优化后的 query 作为基础
- `search-policy.json` 配置：`searchEnhancement.selfImprovingStrategy`

---

### 不确定性自动查验规范（P0 规范）
**行为**：写稿中如果存在不确定性（事实不清、数据未确认），须自动触发联网查验
**限制**：联网查证充分性阈值 < 0.8 时，**须**触发补查

> 不确定性查验来源于书稿写稿不确定内容，须联网查验

**规范**：
```
不确定性自动查验规范
执行 WebSearch 查验，query = {不确定内容关键词}（联网查验不确定内容是否有明确资料）
查验结果摘要：充分 / 不确定内容仍不清楚（须标注）
```

- `search-policy.json` 配置：`searchEnhancement.autoSearchOnUncertainty`

---

### 来源质量追踪规范（P1 规范）
**行为**：联网查证来源须进行质量追踪，高质量域名须记录，低质量来源须降级
**规范**：
1. **S0 查证**：执行 `[S0] 联网查证` 表格，高质量来源须记录（最近 3 次）
   | 来源标题 | URL | 查证时间戳 | 质量等级（L1/L2） |
   |---------|-----|---------|--------------|

2. **高质量域名**：S3 写稿中高质量来源须追踪，优先使用 `.fbs/high-quality-domains.json` 中已记录的高质量域名，优先 WebFetch 直接读取
3. **来源质量记录**：联网查证结果若来源质量低，须在结果摘要中标注，并降级处理，不用于核心论据
4. **`search-policy.json` 配置**：`searchEnhancement.sourceQualityTracking`，`searchEnhancement.searchSourceDiscovery`

---

### 联网查证汇总表

| 规范 | 级别 | 适用阶段 | 主要作用 |
|------|------|---------|-----------|
| 年份时间戳查验 | **P0** | S0 查证 | 确认查证结果年份准确性 |
| 年份硬编码禁止 | **P0** | 写稿全程 | 禁止无来源年份直接写入 |
| 关键词时效注入 | **P0** | S3 写稿联网 | query 携带年份，降低过期数据风险 |
| query 优化规范 | **P1** | 写稿查证 | 精准化 query，记录 ledger |
| 不确定性自动查验 | **P0** | 写稿全程 | query 包含不确定内容，联网查验 |
| 来源质量追踪 | **P1** | S0 查证 + S3 写稿 | 标注高质量来源，优先使用 S0 高质量3条 |

## §3.0.7 联网查证 ledger 记录规范（P0 强制，v1.6.5 新增）
> **来源**：某项目出现联网查证记录丢失，`search-ledger.jsonl` 字段不完整，后续 ledger 无法追溯，规范修订后须严格执行

### 记录要求
所有 WebSearch / WebFetch 查证结果**必须完整记录**到 `.fbs/search-ledger.jsonl`（`.fbs/` 须在书稿根目录，`.fbs/` 目录须在用户工作区根目录）
```jsonl
{"timestamp":"ISO","query":"查询词","url":"来源URL","ok":true,"resultSummary":"约200字摘要","chapterId":"S0/ChNN/report","depthLevel":"L1/L2/L3"}
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | 必填 | ISO 日期时间戳 |
| `query` | 必填 | 联网查证 query |
| `url` | 必填 | 查证来源 URL，若无来源则留空 |
| `ok` | 必填 | 查证是否成功（true/false） |
| `resultSummary` | 必填 | 约200 字摘要 |
| `chapterId` | 必填 | 章节标识符，`S0`、`Ch01`、`report` |
| `depthLevel` | 必填 | 查证深度分级：`L1`（摘要层）/ `L2`（深度层）/ `L3`（验证层） |

> **字段别名兼容（v1.7.0 新增）**：允许使用别名：`ts`=`timestamp`，`summary`=`resultSummary`，`chapter`=`chapterId`，`depth`=`depthLevel`，别名须统一规范化，脚本验证：`node scripts/enforce-search-policy.mjs --skill-root <技能根> --book-root <书稿根> --chapter-id <id> --verify-atomicity`

### 执行规则

- 若 `search-ledger.jsonl` 不存在，须创建（`timestamp` 确保 WebSearch/WebFetch 查证记录完整记录）
- **A 类书稿**：S3 联网查证每章必须记录
- **B 类书稿**：联网查证须尽量完整记录
- **C 类书稿**：联网查证记录可选

### B/C 类记录要求
- 若 `.fbs/` 不存在，B/C 类书稿联网查证须**尽量完整记录**，禁止**完全不记录联网查证结果**
- 记录格式：`★ 查证记录 [query] 层 [L1/L2/L3] 级 [ok/fail] 摘 [约200字]`

## §3.0.8 联网查证深度分级规范（P1 规范，v1.6.5 新增）
> **来源**：某项目联网查证深度不足，L1（摘要层）查证覆盖率过高，L2/L3 深度查证不足，覆盖深度不足，导致书稿中核心论据不充分

### 深度分级说明
| 深度级别 | 执行方式 | 适用场景 | 推荐使用 | 工具 |
|---------|------|--------|---------|---------|
| **L1 摘要层** | WebSearch 摘要查证 + 汇总 | 泛泛信息、背景知识 | 不推荐单独使用 | WebSearch |
| **L2 深度层** | WebFetch 全文读取深度查证 | 核心论据，深度文献读取 | 推荐核心论据使用 | WebFetch |
| **L3 验证层** | 多 WebFetch 验证 L2 内容 | 核心论据二次验证 | 核心论据必须 | 多次 WebFetch |

### 书稿类型深度要求
| 书稿类型 | 核心章节 | 关键论据 | 背景知识 |
|---------|----------------|----------------|----------------|
| **A 类（叙事/非虚构飞）** | L2 | **L3** | L1 |
| **B 类（报告/行研/分析）** | **L2** | L2 | L1 |
| **C 类（简短/文档）** | L1 | L1 | L1 |

### 深度分级执行逻辑

```
联网查证深度分级执行逻辑（WebSearch 可用时）
  检查 触发联网查证，按照书稿类型/场景/论据等级判断
    若 背景知识 = L1
    若 核心论据，须 WebFetch 读 L2
  检查 触发联网深度核实，来自核心论据/案例/数据/调研
    若 是 A 类书稿
      执行 深度查证 L3（2 次以上 L2 后执行 L3 深度验证）
    若 是普通论据/背景
      执行 L1 摘要查证
```

### 深度记录

- `search-ledger.jsonl` 中 `depthLevel` 字段须标注深度分级
- Report Brief / Chapter Brief 联网查证摘要须包含深度分级信息
- 禁止 L1 摘要查证单独作为核心论据，须有 L2 深度查证佐证

> **来源**：某书稿第5-58章章节数量偏多：600章以上，导致25章以上 AI 写作模式明显，书稿写作后期质量不可控 
> **建议**：联网查证规范须防止书稿后期 AI 写作模式，防止堆砌
**联网查证规范**：
| 规范 | 说明 |
|------|------|
| **AI 写作禁止** | 禁止书稿中使用 AI 写法模式（-3 分以上），禁止连续几章只有表格、列举，没有叙事内容 |
| **联网防堆砌** | 联网查证须联网查证核心论点，防止书稿联网查证结果堆砌，仅用联网查证内容填充书稿，不结合叙事 |
| **联网查证复用** | 联网查证结果须跨章节共享，防止同主题重复查证，降低 AI 写作模式 |
| **叙事优先** | S2 目录规划阶段，书稿写稿要求叙事优先，每章叙事优先，防止 AI 写法 |

**联网查证防堆砌模板（骨架检测）**：
```markdown
S3 写稿中期检查，进入第N章时，是否存在联网结果堆砌？
当前第N章整体评分：{当前章节/10}
书稿写法是否偏 AI 写作模式？
优化建议：
A. 叙事方式重写第N+1章（推荐）
B. 回顾前3章联网查证结果是否已经内化
C. 暂停继续查证，叙事化写法
D. 查看书稿写法分析建议
选 A/B/C/D 或输入其他需求（默认 A）
```

### S3 联网查证规则（工具调用超时与速率限制处理）

> **说明**：联网查证每次 query 超时约 15s，单次查证若超时则记录结果，**建议每次不超过单次** query 查证 15s 超时，**工具调用超时** — Read 工具调用超时，**合理处理联网查证超时**，联网查证超时/限速 **`too many requests`（WorkBuddy Error Code 14003）**，需要按照下列规则处理超时与限速

| 情况 | 处理方式（联网查证工具调用限制） |
|------|--------------------------------|
| 单次 超N 章联网查证 | **超时处理规则**，联网查证次数，S3 联网查证禁止超过 **联网超时N 秒**，N 见 `search-policy.json` 超时设置，禁止超时继续等待 |
| 超时30s继续等待 | 联网查证超时规则，**超时30s** 触发超时处理，超时时告知 `SKILL.md` 第10 **超15s** 超时联网，查证降级，每次1/2次超时，继续查证后 `webSearch` 工具降级15s 执行，注意*15s 超时**覆盖**，等待**超时**，继续**写稿**，**联网查证UI 最终执行** |
| **触发14003 / too many requests** | **多次触发 QPS**，触发速率限制后，退避等待 N 秒，N = `rateLimitGuard.backoffOnRateLimitMs`，默认10s，退避后 **继续写稿**，触发超过 `maxRetriesOnRateLimit` 时，默认2 次，记录 `ok=false reason=rate_limit` 到查证记录，禁止无限重试，**速率限制记录**：S0/S3 使用 WebSearch 联网查证时限制单次 QPS 超限最大限制，避免超限 |
| 规范不一致 | **禁止跳过不读取规范**，联网查证超时后，**Read 工具 3 次以内**，再次调用，查阅 [`skill-index.md`](./skill-index.md) 规范查证再读取 |
| 超时降级 | **降级**：仰角联网超时后，降级为模型知识，告知用户降级写稿 |

#### 触发 14003 / too many requests 标准处理流程
> 适用于 **WorkBuddy** 宿主出现 QPS 限速触发

| 步骤 | 说明 | 操作 |
|------|------|------|
| 1 | **检测到限速触发** | 停止继续发送 WebSearch，记录触发时间戳 |
| 2 | **进入退避等待** | 等待 10s（`rateLimitGuard.backoffOnRateLimitMs`），等待期间禁止写稿 |
| 3 | **检查间隔时间** | 确保间隔 Xs（`minIntervalBetweenQueriesMs`）后才继续发送 |
| 4 | **超过重试次数** | 超过 `maxRetriesOnRateLimit` 次数（默认2）时，停止重试，记录 `rate_limit` 到查证记录 |
| 5 | **降级写稿** | 告知用户已触发 QPS 限速，写稿 X/N 章已完成，Y 章将使用模型知识替代联网查证 |
| 6 | **更新配置** | 确认 `search-policy.json` 的 `searchAccessPolicy.s0MaxConcurrentQueries` 并发限制为2，S0 联网并发查证限制，不超过并发限制 |

**实现参考**：`BookWorkflowOrchestrator.execute({ mode: 'chapter_gate', 以及 onProgress })` 和 `chapter_pipeline` 管道均已内置 **退避策略**，参考实现见 `SearchBundle.executeChapterGate` / `MultiAgentPipeline`，负责跟踪 `nextAction`，`gateQueryIndex` 继续查证断点，中断后可恢复，`rateLimitGuard` 自动累计退避等待

#### S3 写稿中联网查证出现 14003 的处置规则
> **背景**：S3 阶段**章节写稿**中途出现限速，若一直等待会大量浪费 token、用户时间，且 WorkBuddy 出现 `too many requests (Error Code: 14003)` 
> **策略**：采用最小中断写稿策略，尽量减少中断时间，最大化利用 token，避免 QPS 触发

**两类常见问题与对策**

| 问题 | 对策 |
|------|------|
| **每章联网查证超5s 时继续等待** | 采用超时记录策略，超过时间后不等待5s 继续下一章，章节联网查证配额2次，3次后降级，继续执行以内2/3章等待，继续超时以内重试 → 降级 ≤5s 继续超时**降级执行** |
| **联网查证超时整章节暂停** | 禁止对 `###` 节点超时00超时00次，超时触发导致章节 token 超过，禁止等待过长超时 |

**最大化写稿连续性原则**

对超过**联网查证 15～0 秒**的写稿，最大化原则是连续写稿，记录超时到查证记录但不让超时中断写稿，超时后继续写稿

**联网查证超时梯度处理表**

| 情况 | 处理方式 | 说明 |
|------|----------|------|
| 联网查证超时触发限速 | 中断 N 秒后，继续写稿 [下一章继续写稿]，不要重复等待，| 联网查证 Read 工具调用超时，**降级**联网查证，**继续**写稿，**降级**联网查证 |
| 单章限速，同时完成其他 | 中断 N 秒后进行 N 章查证 N 次1重试 | 超时后降级写稿，不再查证，参考 S0 联网查证 |
| 重复限速出现 | 触发后暂停写稿，告知用户继续 | 执行脚本（`sync-book-chapter-index.mjs`）继续 |

**联网查证超时注意**：
- 超时处理后必须继续写稿，不得无限等待
- 禁止超时后无限重试联网查证
- 超时后写稿改用 S0 联网模型知识降级

**限速触发后恢复 14003 处置流程**

| 触发场景 | 恢复策略 |
|----------|------|
| 单次 30s超时触发 14003 | 等待 30~60s 后恢复写稿，继续写稿 |
| 多次超时 | 超时进入 WorkBuddy 限速等待，暂停写稿 |
| 写稿降级 | 联网查证降级，继续写稿，使用模型知识补充 |

#### S3 大规模写稿自动断点·Auto-Run Unit 机制

> **使用场景**：大规模书稿超过阈值时，自动触发断点机制，防止单次 token 超限影响章节连贯性
> **执行单位**：以章节 Part 为单位断点，**避免单次处理过多章节一次性提交**

**断点触发条件（M2 项目必须启用，见 `large-scale-book-strategy.md §4.1`）**

写稿进入 Auto-Run 前须满足：
1. 恢复书稿 `.fbs/book-context-brief.md` 上下文记录
2. 加载台账 `.fbs/chapter-status.md` 进行记录，`restoreStateFromDisk` 调用
3. 检查 `.fbs/rate-budget.json` 速率预算，rateLimitRate > 0.3 时触发警告
4. 检查并恢复 `emergency-draft-*.md` 草稿文件，若存在则读取恢复

**Auto-Run 执行单位停顿条件**

| 条件 | 处置 |
|------|------|
| 单章超时5s | 记录超时，继续写 S3 下一章 |
| 写稿异常 | 记录异常 + 更新 `.fbs/chapter-status.md` + 保存紧急草稿 |
| 书稿结构变更 | **暂停写稿**，等待用户确认 Auto-Run 重启（见 `large-scale-book-strategy.md §1b`） |
| 连续写稿超过 20 章 | **必须暂停**，向用户报告进度、请求确认继续，记录到 `.fbs/structure-arbitration-log.md` |
| 超限 rateLimitRate > 0.6 | **强制暂停**，向用户报告速率已触发{rateLimitRate}，建议等待 1分钟后继续 |
| 连续 3 次写稿均失败 | **停止**，向用户报告，等待用户干预 |

**Auto-Run 进度汇报格式**

| 情况 | 输出格式 |
|------|----------|
| 完成一批 | 完成 X 章，已完成 N 章，当前进度写入 .fbs/auto-run-unit-report.md，等待继续 X+1 批 |

### §3.3 写稿断点检查机制（P0 强制，v1.9.0 新增）

#### §3.3.1 写稿断点恢复（P0 强制）
> **说明**：每章写稿完成后须同步 `chapter-status.md` 台账，暂停/恢复写稿时须先恢复台账状态，否则会发生章节重复/遗漏

**断点恢复（P0 强制要求）**：

        若             章节字符数明显不足 < 目标 60%，触发 §3.3.2

#### §3.3.2 骨架检测与字符数不足警告（P0 强制）

      禁止自动补写骨架，需等待用户确认

#### §3.3.4 台账字符数强制实测（P0 强制，2.0.1 新增）
> **来源**：2026-04-02 台账 `chapter-status.md` 中记录的字数仅凭估算约"~4千字"等占位表述，最终实际字数差异导致 S6 骨架误判 4 章，进入 S6 转化/发布准备时误判"166页"实际只有 35~40 页
> **措施**：从台账字符数角度强制实测，通过读取实际文件字符统计，不依赖估算

**台账实测（P0 强制要求）**：
```
台账字符数实测要求
1. 每章写稿完成后，必须实测字符数：使用 read_file 读取对应章节文件
   台账记录必须是实测字数，禁止
   
   错误示例：| Ch01 | 章标题 | ~8000字 |     估算占位
   正确示例：| Ch01 | 章标题 | 8,234字（实测）|
   
2. 若无法读取实测字符数，禁止写入 ~"估算字数
   S5 审校时必须实测字数
   必须 read_file 读取实际文件字符数
3. 若 PowerShell 可用，允许使用 Node/Shell 脚本实测
   推荐使用脚本实测章节字数（Windows PowerShell兼容）
   
   Get-ChildItem "{路径}" -Filter "*.md" | 
     Where-Object {$_.Name -match "S3"} | 
     ForEach-Object { 
       "$($_.Name): $((Get-Content $_.FullName -Raw -Encoding UTF8).Length)字符"
     }

4. 台账格式（2.0.1 强制标注实测）   - 台账字符数{N}字（实测）而不是 {N}字（估算）
   - 汇总：{N}章，字数合计：{M}字，{比例}%   - 禁止提报估算字数，不可以写台账

  ↓ 调用 Proofer（全字段审校）+ 跨章记录 + 结尾钩子
  ↓ 写稿结尾 → Writer完成
  ↓ 评分 ≥7.5 章节通过
  ↓ 评分 7.0～7.5 通过警告，建议润色
  ↓ 评分 <7.0 触发重写规则（2次上限）
      若 2 次重写后 <7.0 强制暂停，等待用户确认

```

### §3.5.1 S5 数据新鲜度必报规则（P0 强制，2.0.1 新增）
> **来源**：2026-04-02 某大数据项目在 S5 审校结束后，"最新数据"标注了 2020 年，S5 审校未覆盖数据时效性检查，导致交付后书稿有 ~5 年时效缺口，20% 章节数据已过期，被投诉误导读者
> **教训**：数据密集型书稿（含统计数据/调研数据/事件时间线），S5 审校必须覆盖数据新鲜度，不得跳过

**数据新鲜度必报（P0 强制）**：
```
S5 审校结束时，强制输出数据新鲜度表（P0 强制）：

════════════════ → S5 审校 ════════════════
数据新鲜度总表（{M}章，含数据引用章节：{N}章）
数据截止时间：{ceil(M÷0.2)}章，达标（80%以内）：{N}章，达标率：{N/M×100}%
数据新鲜度分布（高时效性风险）：

  各章需注意：  时效风险章节清单（需标注更新）：
  如N < ceil(M÷0.2)，表示有不足20% 章节  存在过期数据风险，建议检查最新数据：read_file 重新确认数据出处
```

**数据新鲜度章节格式模板**：
```
数据新鲜度：[S3-ChNN] 章N.md | 数据来源：{数据来源名称} | 数据时效：{数据截止时间/标注来源}
```

**数据时效性分级阈值**：
| 时效级别 | 适用场景 | 数据新鲜度要求 |
|-----------|------------|
| 低时效 | 历史类 | 历史数据不要求100% |
| 中时效（1~5年） | 通用商业/科普 → 数据年份不超过5年 |
| 高时效（1~3年） | 11~20章，数据年份不超过3年 |
| 极高时效 | 21~30章，不超过 ceil(M÷0.2) 章 |
| 最高时效 | 超31章 | 超0章内时效风险 → 全量1章 |

> **时效说明**：超时效风险章节1章后，推荐重新确认来源→ 1次数据声明

# S6 分卷说明

> 本分卷面向 S6 阶段。目标不是把作品丢出书房，而是把书稿从**虚拟书房的完成区**推进到**虚拟书房的转化区**，并在需要时继续推进到**发布区**，形成可传播、可产品化、可外部使用且可回流修订的成果。

## S6 目标

- 明确本轮转化目标。
- 沉淀可复用的转化资产，并在需要时关联 S5 交付物。
- 把转化结果映射回资产面板。
- 在发布区记录外部使用状态与反馈闭环。
- 给出下一步执行动作。

## 必备输入

- 已完成的章节状态台账。
- 已通过的质量检查结果。
- 当前版本的发布范围说明。

## 必备输出

- `[S6]-content-units.md`：内容单元清单。
- `[S6]-product-roadmap.md`：产品化路线图。
- `story-bank.md`：故事库归档状态更新。
- `sessions-summary.md`：追加 S6 决策记录与后续动作。
- `[S6]-release-map.md`：发布映射内部台账（按需，但进入发布区时必须生成）。
- `releases/<chapterId>-release.json`：发布区初始清单，记录待发布状态、交付物引用与渠道信息。


## 执行顺序

1. 读取 `chapter-status.md` 与 `.fbs/chapter-status.md`。
2. 核对完成度、风险项、未闭环事项。
3. 生成内容单元与路线图草案。
4. 更新 `story-bank.md` 归档状态，并向 `sessions-summary.md` 追加本轮 S6 决策摘要。
5. 复核后将内部转化记录落盘到 `.fbs/`，将对外交付物保持在 `deliverables/`。
6. 若存在可外用成果，补做发布映射，并同步写入 `releases/` 的发布区初始状态。
7. 发生真实外部使用后，按 `待发布 → 已发布 → 反馈中 → 回流修改` 四态更新发布区记录。

## 通过标准

- 状态结论清晰。
- 输出文件齐全。
- 发布状态可追踪。
- 风险与后续动作可执行。
- 若发生反馈回流，能明确回到转化区或写作区的后续动作。
