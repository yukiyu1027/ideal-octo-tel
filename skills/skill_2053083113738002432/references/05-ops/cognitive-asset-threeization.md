# 认知资产“三化”与商业机制对齐（v2.1.1）

> 目标：把愿景、文档承诺、代码实现统一到一个可执行口径。  
> 适用：产品规划、Skill 设计、宿主接入、发布与运营复盘。

---

## 1. 统一口径

AI 时代的书，不是一次性文本，而是可持续运营的认知资产。  
统一定义为“三化”：

- **可进化**：每轮写作与改写都产生可追溯增量，而非覆盖式重写。
- **可分发**：同一知识资产可按场景、通道、角色差异化交付。
- **可增值**：资产可通过会员机制与能力包形成持续价值回收。

## 3. 三层一致性要求

### 3.1 愿景规划层

- 所有对外表达应统一为“认知资产三化”，避免只讲“写作效率”。
- 路线图评审至少回答三个问题：
  - 本轮增强如何提升“可进化”？
  - 本轮增强如何提升“可分发”？
  - 本轮增强如何提升“可增值”？

### 3.2 文档承诺层

- `SKILL.md`：入口速查必须可触发三化相关动作（如拆书改写入口、计划模板）。
- `PROJECT-WHITEPAPER.md`：需描述三化模型与商业机制映射。
- `fbs-value-one-pager.md`：需用非技术语言讲清用户价值与升级路径。

### 3.3 代码落地层

- `fbs-runtime-hints.json`：保留 `cognitiveAsset` 机读块，供首响与宿主展示消费。
- `intake-router.mjs`：首响输出 `firstResponseContext.cognitiveAssetSnapshot`。
- 契约测试：确保上述字段在运行时可见，并与 hints 对齐。

## 5. 风险与边界

- 三化是运营口径，不替代质量门禁；发布结论仍以门禁脚本与审计结果为准。
- 乐包为本地激励点，不等同现金支付能力；对外沟通需避免误导。
- 离线模式下增强能力可能降级，但不应影响基础写作与恢复链路。

## 6. 高维洞察：三化与全仓机制的坐标对齐

本节回答「三化」在工程与运营中的**落点**：与阶段流水线、策略 A/B/C、质量门禁、首响机读的关系，避免把愿景当成门禁、或把脚本通过当成战略完成。

### 6.1 与 S0–S6（ESM）的关系

| 三化 | 在阶段流水线中的读法 |
|---|---|
| **可进化** | 各阶段门禁、台账、`session-exit` 与恢复链，使改动可追溯、可接续，而非「覆盖即忘」。 |
| **可分发** | 场景包坐标、多通道 manifest、体裁路由，使同一套技能按宿主与场景差异化交付。 |
| **可增值** | 乐包与离线/在线分层，把「能力解锁」与「可持续使用」产品化，而不是单次对话即弃。 |

### 6.2 与策略 A / B / C 的关系

见 [`memory-layer-matrix.md`](../01-core/memory-layer-matrix.md) 与 `fbs-runtime-hints.json` → `strategiesABC`：

- **A（真值与契约）**：为「可进化」提供磁盘与契约基础；没有 L1 真值与 intake 契约，资产演进无法审计。
- **B（宿主增益）**：支撑「可分发」与「可增值」在宿主侧落地（能力探测、镜像、协作通道等）。
- **C（演进治理）**：词表治理、升级摘要等，对齐长期品牌一致性与资产增值叙事。

### 6.3 与质量门禁的关系（边界声明）

- **三化**是价值叙事与路线图语言；**S/P/C/B、各 gate、`audit:*`** 是合格性与一致性语言。
- 叙事不得替代门禁结论；门禁通过也不得偷换三化层面的对外承诺（例如仍须诚实披露离线降级与乐包非现金）。

### 6.4 与首响、宿主展示的关系

- 用户可见的愿景一句话，应以 `fbs-runtime-hints.json` → `cognitiveAsset.userValueOneLinerZh` 及首响 `firstResponseContext.cognitiveAssetSnapshot` 为锚。
- 宿主「关于 / 价值说明」若引用愿景，应消费上述机读字段或 `strategyDoc`，避免与文档、发版说明分叉出第三套表述。
